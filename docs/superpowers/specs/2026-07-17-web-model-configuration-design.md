# Web Model Configuration and Safe Embedding Switching Design

**Date:** 2026-07-17

## Goal

Make ScholarAgent's deployed web console the primary place to configure and verify the Agent LLM and Qwen embedding service. The flow must work through the Docker frontend at `http://localhost:3000`, keep secrets out of read responses and logs, and prevent retrieval from mixing vectors generated by different embedding models.

## Current State and Problems

- The legacy console already contains fields for LLM and embedding configuration, and the backend persists deployment-wide runtime settings in PostgreSQL.
- The Docker nginx configuration proxies only a few route prefixes. Login currently posts to nginx static hosting and returns HTTP 405, so the settings UI cannot be reached through the deployed frontend.
- The UI still advertises removed MySQL, JSON, lexical-provider, BGE, Jina, and Cohere storage/embedding choices. The backend now requires PostgreSQL, pgvector, and the Qwen embedding adapter.
- The LLM probe only tests the already-saved configuration. There is no way to test candidate values before activation and no dedicated embedding probe.
- Changing the active embedding model would leave old vectors searchable alongside new vectors. A vector column with the same dimensions does not make vectors from different models comparable.

## Scope

### Included

- Repair the Docker frontend-to-backend API gateway.
- Align the web settings UI with PostgreSQL/pgvector and Qwen-only embedding constraints.
- Configure primary and fallback Agent LLM providers, endpoints, keys, and model names.
- Probe unsaved candidate LLM and embedding settings before activation.
- Persist deployment-wide settings in PostgreSQL with masked secret readback.
- Activate a new embedding model only after an exact 1024-dimensional probe succeeds.
- Exclude stale or differently-modelled vectors from vector retrieval.
- Provide an explicit re-embedding action and progress/status reporting while lexical retrieval remains available.
- Verify the complete flow with automated tests and direct browser interaction.

### Not Included

- Per-tenant billing, quotas, or multiple named model profiles.
- A general encrypted secret vault or cloud KMS integration.
- Changing the pgvector schema away from `vector(1024)`.
- Automatic provider model discovery from remote vendor APIs.

## Architecture

### Frontend Gateway

Nginx remains the static frontend server and same-origin API gateway. All ScholarAgent API namespaces used by the console are proxied to `backend:8000`; static assets and SPA fallback stay local. Proxy headers and streaming behavior are shared from one include-style configuration to avoid route-by-route drift.

The browser continues to call relative paths. No backend hostname or CORS configuration is exposed to the browser.

### Runtime Model Configuration

The existing PostgreSQL-backed runtime configuration remains deployment-scoped. Only authenticated users with the `tenant_admin` role may read public configuration state, probe candidates, or save settings.

The UI has two independent panels:

1. **Agent model routing** — primary provider, secondary provider, OpenAI-compatible endpoint/key/model, and Anthropic endpoint/key/model.
2. **Qwen embedding** — endpoint, key, model, fixed dimension 1024, active model status, stale chunk count, and re-embedding controls.

Saving a blank secret preserves the current stored secret. Read responses never contain the stored secret and expose only `configured` plus a masked effective value.

### Candidate Probes

Probe endpoints accept candidate settings without persisting them:

- `POST /settings/model/probe` tests the candidate primary model with a minimal prompt.
- `POST /settings/embedding/probe` requests one embedding for a fixed probe string and validates count, numeric/finite values, non-zero norm, and exactly 1024 dimensions.

Candidate secret fields are used only for that request. Exceptions returned to the browser are normalized and redacted.

### Activation and Re-embedding

LLM settings take effect after save because each model call resolves current runtime settings on demand.

Embedding activation follows a stricter state transition:

1. Probe the candidate configuration successfully.
2. Persist the new endpoint/key/model and fixed dimension.
3. Mark ready chunks whose `embedding_model` differs from the new model as `stale` and clear their vector values.
4. Retrieval filters vector candidates by the active embedding model and `embedding_status='ready'`.
5. Lexical retrieval continues throughout the transition.
6. An administrator starts re-embedding for stale/failed chunks. Existing paper ingestion job records track progress and failures.

The system never treats model name changes as a harmless configuration edit.

## API Shape

Candidate values use a dedicated DTO rather than the unrestricted runtime-settings dictionary. Secret values remain optional so a probe can reuse the configured secret when the field is blank.

The public settings response adds an embedding status summary:

- active model and dimension;
- total, ready, stale, and failed chunks;
- whether a secret is configured;
- whether hybrid vector retrieval is currently ready.

Re-embedding endpoints are idempotent: already-ready chunks for the active model are skipped.

## Error Handling and Security

- Keys are never placed in URLs, browser storage, response payloads, traces, or application logs.
- Probe and save routes require `tenant_admin`.
- Unsupported providers, empty remote keys, missing model names, invalid endpoints, and non-1024 embeddings return actionable 422/502 errors.
- Blank secret input means preserve, not delete. Secret deletion is a separate explicit action outside this scope.
- A failed probe leaves persisted configuration unchanged.
- A failed re-embedding job records a redacted error while lexical content remains searchable.

## UI Behavior

- The login flow works through the Docker frontend.
- Removed storage choices are displayed as fixed PostgreSQL/pgvector facts, not editable controls.
- Provider presets fill sensible endpoint placeholders but remain editable for gateways such as OneAPI, LiteLLM, Ollama, vLLM, and LM Studio.
- Each panel has its own `Test connection` action and structured result area.
- Save buttons show pending/success/error states and never repopulate password inputs.
- Embedding model changes show a re-index warning before activation and a visible lexical-fallback status until re-embedding completes.

## Testing

- Unit tests for secret preservation/masking, candidate DTO validation, probe behavior, active-model vector filtering, stale transitions, and idempotent re-embedding selection.
- Route tests for authorization and redacted errors.
- Static/config tests for nginx proxy coverage and removal of legacy backend choices.
- Docker verification for nginx, backend health, PostgreSQL migration, and frontend access.
- Browser verification: login, navigate to settings, inspect model and embedding panels, validate candidate errors without persistence, save non-secret fields, refresh, and confirm persistence/status.

Real provider keys are not required for automated tests. Live paid-provider probes are performed only when the user supplies and authorizes a real key.
